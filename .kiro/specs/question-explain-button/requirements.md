# Requirements Document

## Introduction

This feature adds a "Learn More" button to the BowlPrep practice interface. When a student is viewing a question, they can click the button to receive a deeper explanation of the topic covered by that question. The explanation is generated on-demand by an AWS Bedrock language model, invoked server-side via a new Next.js API route. The feature integrates into the existing `QuestionWorkspace` component and follows the same server-side AWS access pattern already established by the `/api/categories` and `/api/question` routes.

## Glossary

- **App**: The BowlPrep Next.js web application.
- **API_Route**: A Next.js server-side API route handler.
- **Question**: A Science Bowl question record as returned by `/api/question`.
- **QuestionWorkspace**: The UI region that displays the current Question and collects the student's answer.
- **ExplainButton**: The "Learn More" button rendered inside the QuestionWorkspace.
- **ExplanationPanel**: The UI region that displays the Bedrock-generated explanation for the current Question.
- **Bedrock_Client**: The AWS SDK v3 Bedrock Runtime client used exclusively in server-side API routes.
- **Explanation**: A plain-language, topic-focused description generated by the Bedrock model in response to a Question's content.
- **ExplainRequest**: The JSON body sent from the browser to `POST /api/explain`, containing the question fields needed to construct the Bedrock prompt.

## Requirements

### Requirement 1: Learn More Button

**User Story:** As a student, I want a "Learn More" button on every question, so that I can get a deeper explanation of the topic when I'm unsure or curious.

#### Acceptance Criteria

1. WHEN a Question is displayed in the QuestionWorkspace, THE QuestionWorkspace SHALL render an ExplainButton labeled "Learn More".
2. WHEN the ExplainButton is clicked and an explanation fetch is in progress, THE ExplainButton SHALL be disabled and SHALL display a loading indicator in place of the "Learn More" label text.
3. WHEN the ExplainButton is clicked and an ExplanationPanel is already visible for the current Question, THE App SHALL hide the ExplanationPanel so it is no longer visible.
4. WHEN a new Question is loaded into the QuestionWorkspace, THE App SHALL hide any previously displayed ExplanationPanel and restore the ExplainButton to its enabled state with the "Learn More" label.
5. IF an explanation fetch is in progress when a new Question is loaded, THE App SHALL cancel the in-flight request and restore the ExplainButton to its enabled state with the "Learn More" label.

---

### Requirement 2: Explanation API Route

**User Story:** As a developer, I want a dedicated API route that generates a topic explanation for a given question, so that the frontend can request explanations without embedding AWS credentials or Bedrock SDK calls in browser-executed code.

#### Acceptance Criteria

1. THE API_Route at `POST /api/explain` SHALL accept a JSON request body containing: `question_stem`, `answer`, `Category`, and `MatchType`.
2. WHEN a valid ExplainRequest is received — where all four required fields are present and non-empty strings — THE API_Route SHALL construct a prompt that includes the `question_stem`, `Category`, `MatchType`, and `answer`, then invoke the Bedrock_Client to generate an Explanation.
3. WHEN the Bedrock_Client returns a successful response, THE API_Route SHALL return a 200 JSON response with an `explanation` field containing the generated text.
4. THE API_Route SHALL invoke the Bedrock_Client exclusively server-side; no Bedrock SDK calls or AWS credentials SHALL appear in browser-executed code.
5. IF the App is running locally, THEN THE API_Route SHALL initialize the Bedrock_Client using the `onasmmon` AWS profile.
6. IF the App is running in production, THEN THE API_Route SHALL initialize the Bedrock_Client using the attached IAM role without referencing any named profile.
7. IF the request body is missing any required field or any required field is an empty string, THEN THE API_Route SHALL return a 400 JSON response with a `message` field that names each missing or empty field individually.
8. IF the Bedrock_Client invocation fails, THEN THE API_Route SHALL return a 500 JSON response with a `message` field; the response SHALL NOT include a stack trace or internal error details.

---

### Requirement 3: Explanation Display

**User Story:** As a student, I want to read the generated explanation in a clear panel below the question, so that I can learn the topic without leaving the practice page.

#### Acceptance Criteria

1. WHEN the API_Route returns a successful explanation, THE App SHALL display the ExplanationPanel below the QuestionWorkspace containing the `explanation` text.
2. WHILE the explanation fetch is in progress, THE App SHALL display a skeleton or spinner element in the ExplanationPanel region to indicate loading.
3. WHEN the ExplanationPanel is visible, THE ExplanationPanel SHALL pass the explanation text to the existing `MathText` component for rendering, so that LaTeX expressions in the explanation are rendered as formatted math.
4. THE ExplanationPanel SHALL include a dismiss button; WHEN the dismiss button is clicked, THE App SHALL hide the ExplanationPanel and restore the ExplainButton to its enabled state with the "Learn More" label.
5. WHEN the ExplanationPanel is visible and the student submits an answer, THE App SHALL keep the ExplanationPanel visible.
6. WHEN the ExplanationPanel is visible and the student submits an answer, THE App SHALL NOT hide or reset the ExplanationPanel.

---

### Requirement 4: Error Handling for Explanation Fetch

**User Story:** As a student, I want a clear error message if the explanation fails to load, so that I know something went wrong and can try again.

#### Acceptance Criteria

1. IF the `POST /api/explain` request returns a non-2xx HTTP status, THEN THE App SHALL display an error message in the ExplanationPanel region stating that the explanation could not be loaded.
2. IF the `POST /api/explain` request fails due to a network error or thrown exception, THEN THE App SHALL display a generic error message in the ExplanationPanel region rather than remaining silent or showing no content.
3. WHEN an explanation fetch error is displayed, THE App SHALL render a "Try Again" button in the ExplanationPanel region that, when clicked, re-issues the `POST /api/explain` request for the current Question using the same request body.
4. WHEN an explanation fetch error is displayed, THE error message and "Try Again" button SHALL remain visible until the student clicks "Try Again" or a new Question is loaded.
5. IF the "Try Again" request also fails, THEN THE App SHALL display the same error message and "Try Again" button again; THE App SHALL NOT enter a state where the ExplainButton is permanently disabled or the UI is unresponsive.
